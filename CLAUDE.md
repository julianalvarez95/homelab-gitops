# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

GitOps repo for a single-node k3s cluster running on a repurposed Dell
Latitude 7490. ArgoCD watches this repo and applies every change
automatically, with self-heal enabled — nothing in the cluster should be
edited by hand with `kubectl`; if it is, ArgoCD reverts it back to what's
in Git. All work here means: edit YAML/code, commit, push, and let ArgoCD
sync.

Core principle driving design decisions: **LLM inference happens over API
(OpenAI/Claude), the local hardware only orchestrates.** The node has no
GPU and isn't meant to run models locally — agents are lightweight
CronJobs that call out to a hosted LLM, do their work, and exit.

## Repo layout

- `agents/` — each subdirectory is one agent: a CronJob + Dockerfile +
  Python source + its own `kustomization.yaml`. `agents/morning-digest`
  is the reference implementation; new agents should follow the same
  shape (own folder, own Dockerfile, own CronJob).
- `apps/` — reserved for an app-of-apps ArgoCD pattern once there are
  enough agents/services to warrant it. Currently empty.
- `infra/` — reserved for observability tooling (Arize Phoenix,
  VictoriaMetrics) planned for later. Currently empty.

## Building and deploying an agent

There is no CI yet — images are built and pushed manually from the
cluster host, then ArgoCD picks up the CronJob change from Git.

```bash
# from an agent's directory, e.g. agents/morning-digest
docker build -t ghcr.io/julianalvarez95/morning-digest:latest .
docker push ghcr.io/julianalvarez95/morning-digest:latest
```

Kubernetes manifests are applied via kustomize (`kustomization.yaml` per
agent), not raw `kubectl apply -f`, because feed/config files are wired
in through `configMapGenerator`.

`k3s kubectl` does **not** respect `~/.kube/config` by default — it reads
`/etc/rancher/k3s/k3s.yaml` (root-only) unless `KUBECONFIG` is exported
explicitly. Set `KUBECONFIG=/etc/rancher/k3s/k3s.yaml` (or a copy with
sane permissions) before running `kubectl`/`k3s kubectl` against this
cluster.

## Secrets

Secrets are deliberately kept out of Git for now — this is GitOps,
*partial and pragmatic*. Kubernetes `Secret` objects (e.g.
`morning-digest-secrets`) are created directly on the cluster with
`kubectl create secret`, and CronJobs reference them by name via
`envFrom.secretRef`. Do not put credentials in YAML committed to this
repo. SOPS+KSOPS was evaluated and intentionally deferred — revisit it
once there are several agents each with their own secrets, not before.

## Agent pattern (see `agents/morning-digest`)

Each agent is a `CronJob` (not a long-running `Deployment`) — it runs,
does its job, and exits, keeping steady-state RAM usage on the node at
zero between runs. `agent.py` typically:

1. Pulls input from external sources (RSS via `feedparser`, etc.).
2. Calls an LLM API (OpenAI in `morning-digest`) to transform/summarize.
3. Delivers the result somewhere (Telegram via raw `requests.post` to the
   Bot API in `morning-digest`).

Config that isn't a secret (e.g. `feeds.yaml`) is mounted from a
`ConfigMap` generated by kustomize, not baked into the image, so it can
change without a rebuild.

### Known gotchas worth remembering when touching agent code

- **Pin dependency versions in `requirements.txt`.** `openai==1.54.0`
  broke with a `TypeError` about `proxies` because an unpinned `httpx`
  pulled a newer release that dropped an internal param. Fixed by
  pinning `httpx==0.27.2`. Don't leave transitive deps unpinned.
- **Never let API calls fail silently.** The first Telegram delivery
  looked successful (`Completed`, exit 0) but no message arrived,
  because errors from the Telegram API were swallowed. Always log the
  response and call `resp.raise_for_status()` (or equivalent) on
  outbound HTTP calls in agents.
- **Telegram bot tokens from BotFather sometimes include a duplicated
  `bot` prefix** when copy-pasted from the confirmation message
  (`botbot123:...`). `send_telegram()` strips it defensively with
  `.removeprefix("bot")` — keep that in place, and expect the same class
  of copy-paste error from other API tokens.

## Cluster-level notes

- System timezone is `America/Argentina/Buenos_Aires` so CronJob
  schedules (e.g. `schedule: "0 10 * * *"`) fire at the intended local
  time, not UTC.
- SSH access to the node is key-only (ed25519, `PasswordAuthentication
  no`). Be aware that `UsePAM yes` can leave a password bypass via
  `KbdInteractiveAuthentication` even with `PasswordAuthentication no` —
  if touching `sshd_config`, verify with
  `ssh -o PubkeyAuthentication=no` that password login is actually
  rejected, and keep a second session open while making the change.
